## §1 — Context, goals, and architecture approach

Phase A converts the backend from a **single-tenant-per-user** model
(`users.account_id` is the tenancy anchor; `accounts.account_type` is
`:individual | :group`) to a **membership-based multi-tenancy** model where a
join entity, `AccountMembership`, is the sole source of truth for "which User
belongs to which Account, in what role, with what status". A single User can
hold one membership per Account ("multi-familia") and switch the active Account
without re-authenticating.

Goals:

1. Introduce `account_memberships` and migrate all existing `(user, account)`
   pairs into it without downtime or data loss.
2. Replace the `:individual | :group` taxonomy with a richer `Account.plan`
   enum (`:individual | :family_4 | :family_6 | :trial`) resolved through the
   `subscription_plans` table for seat caps and planning-day limits.
3. Ship a **dual-write Guardian** so both the legacy `access` (`access_v1`)
   token and the new membership-scoped `access_v2` token verify simultaneously,
   enabling a zero-downtime, no-forced-re-login cutover gated by a single flag.
4. Deliver the tenancy use cases (invite, accept, roster, remove, leave,
   switch-account, seat-cap enforcement) behind clean HTTP + WebSocket surfaces.

**Architecture approach.** The system follows the existing
Clean/Hexagonal layering already present in `meal_planner_api`:

- **Persistence layer** (`MealPlannerApi.Persistence.Accounts.*`) — Ecto schemas
  are *dumb data*: no tenancy logic, only casting/validation. A dedicated query
  module (`AccountMembershipQueries`) is the single source of truth for the two
  "load the real membership" lookups so the plug, the socket plug, and the
  application context cannot drift.
- **Application layer** (`MealPlannerApi.AccountsMembership`,
  `MealPlannerApi.Services.InviteService`, `MealPlannerApi.Accounts`) — the use
  cases, seat-cap policy, invite token model, and the `access_v2` claim builder.
- **Web layer** (`MealPlannerApiWeb.*`) — plugs (`VerifyTokenType`,
  `LoadCurrentMembership`, `EnforceAccountScope`), controllers, and channels.
  Controllers/channels read `current_membership.account_id` for tenancy, never
  `current_user.account_id`.

The change was delivered as three chained PRs (PR 1 data model + dual-write
Guardian; PR 2 application layer; PR 3 web layer), with the cutover flag held
`false` until PR 3 was deployed. See §9.

## §2 — Data model

### §2.1 — `account_memberships` table

Migration `20260625000001_create_account_memberships.exs` creates the join
table with a `:binary_id` (UUID) primary key:

| Column | Type | Notes |
|---|---|---|
| `id` | `binary_id` | primary key, autogenerated UUID |
| `account_id` | `binary_id` | FK → `accounts`, `on_delete: :delete_all`, `null: false` |
| `user_id` | `binary_id` | FK → `users`, `on_delete: :delete_all`, `null: false` |
| `role` | `:string` | CHECK constrained, `null: false` |
| `status` | `:string` | CHECK constrained, `null: false` |
| `invited_by_user_id` | `binary_id` | FK → `users`, `on_delete: :nilify_all` (audit trail survives inviter deletion) |
| `invite_token_hash` | `:string` | SHA-256 hex of the plaintext invite token; nil once accepted-or-never-invited |
| `invite_expires_at` | `:utc_datetime_usec` | invite TTL boundary |
| `joined_at` | `:utc_datetime_usec` | set on `:invited → :active` |
| `inserted_at` / `updated_at` | `:utc_datetime_usec` | `timestamps/1` |

**CHECK constraints** (named to the project convention):

- `account_memberships_role_check`: `role IN ('owner', 'member')`
- `account_memberships_status_check`: `status IN ('active', 'invited', 'suspended')`

**Indexes:**

- `account_memberships_user_id_account_id_index` on `(user_id, account_id)`
- `account_memberships_account_id_status_index` on `(account_id, status)`
- `account_memberships_user_id_status_index` on `(user_id, status)`
- `account_memberships_active_account_user_unique_index` — **partial unique**
  index on `(account_id, user_id) WHERE status = 'active'`. This is the
  canonical "exactly one active membership per `(account, user)`" invariant; a
  second `:active` row for the same pair raises `unique_violation`.

The `AccountMembership` Ecto schema (`persistence/accounts/account_membership.ex`)
declares `role` and `status` as `Ecto.Enum` (`role: [:owner, :member]`,
`status: [:active, :invited, :suspended]`), `belongs_to :account`,
`belongs_to :user`, and `belongs_to :invited_by, User, foreign_key:
:invited_by_user_id`. The changeset requires `account_id, user_id, role,
status`, validates enum inclusion, applies FK constraints, and binds the
partial-unique constraint by name. **No tenancy logic lives in the schema.**

`:suspended` is a valid enum value but no Phase A code path mints it; it is
reserved for a future re-invitation flow.

### §2.2 — `accounts.account_type` → `accounts.plan` enum + plan seeds

Migration `20260625000002_alter_accounts_to_plan_enum.exs`:

1. Adds `plan :string NOT NULL DEFAULT 'individual'`.
2. Data migration: `UPDATE accounts SET plan = 'family_4' WHERE account_type =
   'group'` (`:group` and `:family_4` are the same seat-cap shape today).
3. Drops the legacy `account_type` column (its CHECK drops with it) — the
   `:group` value MUST NOT exist post-Phase-A.
4. Adds CHECK `accounts_plan_check`:
   `plan IN ('individual', 'family_4', 'family_6', 'trial')`.
5. Seeds two `subscription_plans` rows via idempotent
   `INSERT … ON CONFLICT (name) DO NOTHING`:
   - `family_6` — `max_users: 6`, `max_planning_days: 30`,
     `revenuecat_entitlement_id: "family_6"`
   - `trial` — `max_users: 6`, `max_planning_days: 30`,
     `revenuecat_entitlement_id: "trial"`

   The pre-existing `individual` and `family_4` rows are preserved.

Migration gotcha captured in code: parameterized `binary_id` inserts require the
16-byte form, so the seed uses `Ecto.UUID.dump(Ecto.UUID.generate())`, not the
string form.

The `Account` schema now declares
`field :plan, Ecto.Enum, values: [:individual, :family_4, :family_6, :trial]`,
drops `has_many :users`, and adds
`has_many :memberships, AccountMembership`.

`down/0` re-introduces `account_type` and its CHECK but **deliberately does not
restore data** — a `plan → account_type` mapping would corrupt accounts promoted
to `:family_6`. Real rollback is the coordinated snapshot restore in §9.4.

### §2.3 — `users.account_id` nullable + legacy `users.role`

Migration `20260625000003_make_user_account_id_nullable.exs` relaxes
`users.account_id` to nullable via raw SQL
`ALTER TABLE users ALTER COLUMN account_id DROP NOT NULL` (raw SQL is used
because Ecto's `modify :references` would attempt to recreate the existing FK
and raise `duplicate_object`). This lets `access_v2` holders carry
`current_user.account_id == nil` while `current_membership.account_id` carries
the real tenancy.

The `User` schema drops `account_id` from `validate_required` (keeps the FK
constraint) and adds `has_many :memberships, AccountMembership`. `users.role`
(`Ecto.Enum [:owner, :member]`) is **kept for the dual-write window** — it feeds
the legacy synthesis path historically and the backfill role default. Its later
removal (once `account_memberships.role` is the sole source of truth) is a
follow-up, not part of Phase A (see §10/Q9).

### §2.4 — Backfill `account_memberships` from `users.account_id`

Migration `20260625000004_add_account_memberships_backfill.exs` runs a batched
`DO $$ … LOOP … END $$` block: for every legacy `(users.id, users.account_id)`
pair lacking an `:active` membership, it inserts one `:active` membership with
`role = COALESCE(users.role, 'owner')` and `joined_at = users.inserted_at`.
Batches of **1,000 rows** use `FOR UPDATE SKIP LOCKED` and `pg_sleep(0.05)`
between batches to avoid write-lock storms on a populated production DB. UUIDs
come from `gen_random_uuid()`.

### §2.5 — Invariant function `check_account_membership_invariants()`

The same migration defines a `plpgsql` function
`check_account_membership_invariants()` and invokes it at the end of the
migration transaction. It raises `backfill_invariant_failed` if any of:

1. A legacy `(user, account)` pair (`users.account_id IS NOT NULL`) has no
   `:active` membership.
2. Any `users.account_id` points at a missing Account.
3. Any Account does not have **exactly one** `:owner` `:active` membership.

Because it runs inside the migration's transaction, any violation rolls the
whole migration back — the DB never persists a broken invariant. `down/0` drops
the function; it does not remove backfilled rows (see §9.4).

### §2.6 — Plan → `subscription_plans` resolution (Q10)

Seat caps and planning-day limits are resolved through the `subscription_plans`
table by `name` (not hard-coded in application code):
`Subscriptions.policy_for_account/1` reads `account.plan` and looks up the
matching row. Capacities: `individual → 1`, `family_4 → 4`, `family_6 → 6`,
`trial → 6`. An unknown plan with no matching row returns an error tuple
(`:plan_not_found`). `revenuecat_entitlement_id` may be `nil` for legacy plans —
callers treat `nil` as "no RevenueCat binding", not a fatal error.

> The application-layer seat-cap check (`AccountsMembership.seat_usage/1`) uses
> an in-module `@plan_capacities` map with the same numbers; both agree with the
> seeded `subscription_plans.max_users`.

## §3 — Guardian / JWT: dual-write token strategy

The core of the zero-downtime cutover is a Guardian implementation
(`MealPlannerApi.Auth.Guardian`) that **verifies both token types at all times**
while **minting only one type at a time**, selected by the cutover flag (§9).
The `sub` claim is always `user.id` (`subject_for_token/2`). Guardian sets the
`typ` claim from the `token_type:` option at *encode* time; the application claim
map does not set `typ` for the legacy path, and sets it explicitly for the v2
path (see below).

`resource_from_claims/1` loads the `%User{}` from the DB by `sub` and re-attaches
two fields from the claims for backward compatibility:

- `:subscription_tier` — still read by controllers during the dual-write window.
- `:account_id` — feeds the legacy-token membership lookup in
  `LoadCurrentMembership`.

It **no longer re-attaches `:account_type`** (removed in PR 2b task 2.9): the
legacy `:group | :individual` taxonomy was dropped from the schema and
`Account.plan`, surfaced via `current_membership.plan`, is the source of truth.

### §3.1 — `access` (`access_v1`) claim set

Built by `MealPlannerApi.Accounts.claims_for/2` (`(user, account)`):

| Claim | Value |
|---|---|
| `account_id` | `account.id` |
| `account_type` | legacy string derived from `Account.plan` (compat shim) |
| `subscription_tier` | string (`"free"` / `"premium"`) |
| `email` | `user.email` |
| `name` | `user.name` |
| `linked_user_ids` | list (legacy compat) |

`typ` is **not** in this map — Guardian stamps `typ: "access"` at encode time
via `token_type: "access"`. `sub` = `user.id`, plus `iat`/`exp` added by Guardian.

### §3.2 — `access_v2` claim set

Built by `MealPlannerApi.AccountsMembership.claims_for/2` (`(user, membership)`):

| Claim | Value |
|---|---|
| `typ` | `"access_v2"` (set explicitly in the map) |
| `membership_id` | `to_string(membership.id)` |
| `account_id` | `to_string(membership.account_id)` |
| `role` | `Atom.to_string(membership.role)` (`"owner"` / `"member"`) |
| `plan` | `Atom.to_string(plan)` — string form, e.g. `"family_4"` |
| `status` | `Atom.to_string(membership.status)` (e.g. `"active"`) |
| `email` | `user.email` |
| `name` | `user.name` |

`iat`/`exp` are added by Guardian at sign time (not application claims). The
membership's `Account.plan` is read from a preloaded `:account` when available,
else fetched (avoids N+1 while remaining correct for un-preloaded callers).

### §3.3 — Minting rules and refresh semantics

Which type is *minted* is gated by `tenancy_v2_only?/0`
(`Application.get_env(:meal_planner_api, :tenancy_v2_only, false)`):

- **Flag off (default)** — `AuthController.password/2`,
  `Accounts.authenticate_with_password/1`, `AccountsMembership.accept_invite/2`,
  and `AccountsMembership.switch_account/2` all mint `access` (`access_v1`).
- **Flag on** — the same paths mint `access_v2` using the User's membership via
  `AccountsMembership.claims_for/2`.

`AuthController.issuance_typ/1` returns `:access_v2` only when the flag is on
**and** a real `%AccountMembership{}` came back from the context; otherwise it
falls back to `:access` rather than crashing the v2 claim builder.

**Refresh preserves the original `typ`** and never consults the flag
(`AuthController.refresh/2`): if the incoming refresh token's claims carry a
`membership_id`, the reissued access token is `access_v2`; otherwise it is
`access_v1`. This "no silent re-scoping on refresh" rule holds in both
directions regardless of the flag value at refresh time. `refresh/2` also
asserts `claims["typ"] == "refresh"` explicitly (a missing/other `typ` fails
closed with `401 invalid_refresh_token`) because Guardian does not itself verify
`token_type:` at decode time.

## §4 — Auth pipeline & current resource

### §4.1 — Pipeline order

`MealPlannerApiWeb.AuthPipeline` (the `:auth` pipeline) is:

```
Guardian.Plug.VerifyHeader            # signature + standard claims, NO typ filter
MealPlannerApiWeb.Plugs.VerifyTokenType
Guardian.Plug.EnsureAuthenticated
Guardian.Plug.LoadResource (allow_blank: false)   # → conn.assigns.current_user
MealPlannerApiWeb.Plugs.LoadCurrentMembership      # → conn.assigns.current_membership
```

`VerifyHeader` intentionally carries **no** `claims: %{"typ" => ...}` filter,
because Guardian's `VerifyHeader` can only match a single `typ` value.
`VerifyTokenType` inspects `claims["typ"]` against the supported set
`~w(access access_v2)` and halts unknown values with
`401 {"error":"unsupported_token_type"}`. The supported set is exposed
(`VerifyTokenType.supported_typs/0`) so out-of-pipeline decoders (e.g. the
invite-accept "existing user" path) validate against the same list.

### §4.2 — `LoadCurrentMembership` resolution + `EnforceAccountScope`

`LoadCurrentMembership` reads the decoded claims and resolves
`current_membership` to **a real `AccountMembership` DB row in both cases** —
there is no in-memory synthesized struct (see §11, divergence 1):

- `typ == "access_v2"` → `AccountMembershipQueries.load_membership_by_id/2`, then
  the plug itself requires `status: :active`. A missing/blank `membership_id`, no
  row, or a non-`:active` row all halt with
  `401 {"error":"membership_id_required"}`.
- `typ == "access"` (legacy) → `AccountMembershipQueries.load_active_membership/3`
  for `(current_user.id, current_user.account_id)` with `status: :active`. If no
  such row exists (member removed, or never had one), the request is refused with
  the same `401 membership_id_required`. A removed member's stale legacy token
  (4-week TTL, no server-side revocation) therefore stops working immediately.
- unknown `typ` → same `401` (defensive; the pipeline should already have
  rejected it).

`load_membership_by_id/2` deliberately does **not** filter `status`, because the
Phoenix Channel join path (`LoadCurrentMembershipSocket.membership_from_socket/1`)
needs to distinguish `:invited` at join time and return a channel-specific
`{:error, %{reason: "forbidden"}}`; the socket *connect* must still succeed for a
non-`:active` membership. Active-only fail-closed enforcement lives in the two
callers that have no status-aware step downstream (the HTTP `access_v2` branch
and `AccountsMembership.current_membership/2`).

`AccountMembershipQueries` is the single shared query module for all
current-membership resolution paths (the HTTP plug, the socket plug, and
`AccountsMembership.current_membership/2`) so they cannot drift.

**`EnforceAccountScope`** runs after `LoadCurrentMembership` and only acts on
routes with an `:account_id` path param: it compares
`conn.path_params["account_id"]` against `conn.assigns.current_membership.
account_id` and halts mismatches with `403 {"error":"account_mismatch"}`. Routes
without `:account_id` in the URL (e.g. `POST /api/auth/switch-account`) are a
no-op for this plug.

## §5 — Tenancy scoping model & routing decisions

The central architectural shift: **`membership.account_id` replaces
`user.account_id` as the tenancy anchor.** `current_user.account_id` survives
only as a dual-write compatibility shim (re-attached from the JWT claim for
`LoadCurrentMembership`'s own legacy lookup) and is scheduled for removal once
`access_v1` is retired. Controllers and services MUST read
`conn.assigns.current_membership.account_id`.

Numbered decisions:

- **5.1 — `users.account_id` nullable.** Relaxed to `NULL` for the dual-write
  window so `access_v2` users can carry a `nil` `user.account_id` while the real
  tenancy flows through the membership row (see §2.3).

- **5.2 — Routes + `EnforceAccountScope`.** Six Phase A routes:

  | Method + path | Controller action | Pipeline |
  |---|---|---|
  | `POST /api/accounts/:account_id/invites` | `InviteController.create` | `:api, :auth, :enforce_account_scope` |
  | `POST /api/invites/:token/accept` | `InviteController.accept` | `:api` only (no auth — new-user accept has no token yet) |
  | `GET /api/accounts/:account_id/memberships` | `MembershipController.index` | `:api, :auth, :enforce_account_scope` |
  | `DELETE /api/accounts/:account_id/memberships/:user_id` | `MembershipController.delete` | `:api, :auth, :enforce_account_scope` |
  | `POST /api/auth/switch-account` | `AccountLifecycleController.switch_account` | `:api, :auth` (no `:account_id` in URL) |
  | `POST /api/accounts/:account_id/leave` | `AccountLifecycleController.leave` | `:api, :auth, :enforce_account_scope` |

  The `:account_id`-scoped routes live under a `scope "/api/accounts/:account_id"`
  block. `POST /api/invites/:token/accept` is deliberately **not** behind `:auth`
  because the "new User accepts" case has no account/token yet; the controller
  manually decodes an optional `Authorization` header for the "existing User
  accepts" case, validating `typ` against `VerifyTokenType.supported_typs/0`.

- **5.3 — Plan resolution / `:group` dropped.** Seat caps and limits resolve
  through `subscription_plans` by `name` (§2.6); the `:group` enum value is gone
  from the schema (§2.2). HTTP request bodies still accept the string
  `"account_type"` via the `Accounts.normalize_plan/1` shim (maps `"group"` →
  `:family_4`) so legacy request-level clients keep working.

- **5.4 — Repo query scoping.** The `data/*_repo.ex` modules
  (`account_repo`, `planning_repo`, `inventory_repo`, `shopping_repo`) filter by
  the Account id derived from `current_membership.account_id` rather than
  `user.account_id`. `account_repo.ex` exposes
  `list_active_memberships_for_account/1` (filters `status == :active`,
  `order_by inserted_at asc`, preloads `:user`) for callers that need the
  active-only set; it is explicitly *not* the roster data source (the roster,
  which needs `:active + :invited`, uses `AccountsMembership.list_memberships/1`).
  `get_account_with_users!/1` now preloads `memberships: :user` (the legacy
  `has_many :users` was removed). Multi-familia isolation is regression-locked by
  per-repo tests asserting a User with memberships in Account A and Account B
  cannot read Account B's resources when scoped to A.

- **5.5 — Pipeline accepts both token types.** `VerifyTokenType` accepts
  `access` and `access_v2` so no forced re-login is required across the cutover
  (§4.1).

- **5.6 — `current_membership` is the only trustworthy tenancy source.**
  Routes without `:account_id` in the URL (`/api/calendar`, `/api/planning/*`,
  `/api/cooking/*`, `/api/inventory`, `/api/shopping-*`, …) are not behind
  `EnforceAccountScope`; their isolation is enforced by always reading
  `current_membership.account_id` inside the controller/service/channel.

- **5.7 — Owner cannot leave / cannot be removed.** `leave/2` refuses an owner
  with `:cannot_leave_owned_account`; `remove_member/3` refuses removing the
  owner with `:cannot_remove_owner`. Account transfer/dissolve is a follow-up
  change.

## §6 — Invitation lifecycle (invite → accept)

### §6.1 — `invite/3` transaction + seat-cap enforcement

`AccountsMembership.invite/3` (`(account, inviter_membership, email)`) is
**owner-only** and runs inside a `Repo.transaction/1`:

1. `SELECT … FOR UPDATE` on the Account row (`lock_account_for_invite/1`) — holds
   a row-level lock for the duration to close the seat-cap race.
2. `enforce_seat_cap(account, 1)` — `seat_usage/1` counts `:active + :invited`
   rows and compares `active + invited + 1` against the plan capacity (§2.6).
   Over cap → `{:error, :seat_cap_reached}`.
3. `check_existing_membership/2` — an existing `:invited` row for the email →
   `:already_invited`; an existing `:active` row → `:already_a_member`.
4. `InviteService.create_invite_row/2` inserts an `:invited` membership.

A non-owner actor short-circuits to `{:error, :not_owner}`. On success returns
`{:ok, %{token, expires_at, membership_id, email}}` where `token` is the
plaintext, returned **exactly once**.

**`InviteService` token model** (`services/invite_service.ex`):

- `mint_token/0` → `{plaintext, hash}`: plaintext is 32 bytes from
  `:crypto.strong_rand_bytes/1`, URL-safe base64 no-padding (~43 chars); hash is
  `:crypto.hash(:sha256, plaintext) |> Base.encode16(case: :lower)` (64 hex
  chars). Only the hash is stored.
- `create_invite_row/2` inserts an `:invited` membership with
  `invite_token_hash`, `invite_expires_at = now + 7 days`, `invited_by_user_id`,
  `role: :member`. If the invitee email is unknown a **stub `User`** is created
  (`name: email`, `role: :member`, `password_hash: nil`) to satisfy the FK; it is
  filled in at accept time.
- `verify_and_consume/3` looks up by hash scoped to the account, and returns
  `:invite_token_used` (replay / already consumed), `:invite_token_expired`,
  `:invite_token_unknown`, or `:invite_wrong_account`.

**Accept** — `AccountsMembership.accept_invite/2` has two arities: an existing
`%User{}`, or a new-user map `%{name, password_hash}` (which fills the stub User
via `PersistenceUser.changeset/2` then flips the membership). On accept the row
flips `:invited → :active`, sets `joined_at`, and points `user_id` at the
invitee. Errors: `:invite_token_used`, `:invite_token_expired`,
`:invite_token_unknown`, `:invalid_invitee`.

> **Single-use enforcement (as-built).** The plaintext-hash and
> `invite_expires_at` are **kept on the row** after accept; replay is detected by
> `status != :invited` returning `:invite_token_used`. This diverges from the
> task description that said to null both columns on accept — see §11,
> divergence 4.

`accept_invite/2` and `switch_account/2` build their response claims through
`build_response_claims/3`, which honors the cutover flag: `access_v2` when on,
legacy `Accounts.claims_for/2` when off (see §11, divergence 3). The
`InviteController.accept` HTTP response returns the full auth payload
(`access_token`, `refresh_token`, `user`, `account`, `membership`,
`subscription`, `websocket`).

## §7 — Membership-scoped channels

WebSocket tenancy mirrors the HTTP model but enforces it at **channel join**
rather than through a plug. `UserSocket.connect/3` runs
`Guardian.resource_from_token/1` for `current_user`, then populates
`current_membership` via `LoadCurrentMembershipSocket`. The socket connect
itself must succeed even for a non-`:active` membership so the channel can return
a specific reason.

Each channel topic is `"<channel>:<account_id>"`. `join/3` reads
`LoadCurrentMembershipSocket.membership_from_socket/1` and rejects with
`{:error, %{reason: "forbidden"}}` when the membership is:

- `nil` (no resolvable membership),
- `account_id` mismatches the topic's account id, or
- `status != :active`.

On success it assigns `:account_id` and `:current_membership` to the socket, and
`handle_in/3` callbacks read `current_membership.account_id` for all tenant
scoping (e.g. `CalendarChannel`'s `toggle_favorite` / `upsert_meal`). A legacy
`access_v1` connection resolves to its real `:active` membership and is accepted;
a stale/removed legacy token resolves to `nil` and is rejected. The `:invited`
(non-active) join-rejected behavior is regression-locked per channel.

Channels covered by the sweep: `CalendarChannel`, `PlanningChannel`,
`CookingChannel`, `AiChannel`. **Only these four channels exist on disk** — the
`shopping` and `inventory` domains referenced by earlier planning have no channel
module; their HTTP surfaces enforce isolation via
`current_membership.account_id` in the controller (see §11, divergence 6).

## §8 — Testing strategy & factories

Every task followed strict RED → GREEN → REFACTOR. `:auth`-piped route tests
drive **both** token types (see §8.4). Tests use `start_supervised!/1` (never
`Process.sleep`/`alive?`) and `Req` for HTTP.

### §8.2 — Factory helpers (Q6)

- `user_with_memberships/2` accepts `[{account_attrs, role}, …]` and inserts a
  User + N Accounts + N `:active` memberships, returning the User preloaded with
  `memberships: :account`. Used to build multi-familia fixtures.
- `issue_access_v2_token/2` (`(user, membership)`) mints an `access_v2` JWT via
  `Guardian.encode_and_sign/3` with `token_type: "access"` and the claim map from
  `AccountsMembership.claims_for/2`. (In PR 1 it carried an inline claim builder;
  PR 2a replaced that with delegation to the canonical builder so there is one
  source of truth for the v2 shape.)

### §8.3 — Migration sanity checkpoint

`test/support/migration_sanity_test.exs` (`@moduletag :migration_sanity`) runs a
clean `ecto.drop → create → migrate` from scratch, asserts all four
`subscription_plans` rows exist, seeds pre-Phase-A `users`/`accounts`, runs
`check_account_membership_invariants()` and asserts zero violations, then does a
`rollback → migrate` cycle and re-asserts. It uses `Mix.Task.run/2` only — it
ships no migration files of its own.

### §8.4 — Dual-write token issuance across the `:auth` pipeline

This is the bridge between §3 (JWT shapes) and §4 (pipeline). The Guardian
dual-write test (`test/meal_planner_api/auth/guardian_test.exs`) encodes two JWTs
directly — one `typ: "access"`, one `typ: "access_v2"` (with `membership_id`) —
decodes both, and asserts the claim sets match §3.1 and §3.2 exactly.

At the pipeline level, `:auth`-piped route tests exercise **both** token types:

- an `access_v1` token verifies and reaches the controller;
- an `access_v2` token verifies and populates `current_membership` (a real row);
- an unsupported `typ` (e.g. `access_v3`) halts with
  `401 unsupported_token_type` (`VerifyTokenType`, §4.1);
- an `access_v2` token missing `membership_id` halts with
  `401 membership_id_required` (`LoadCurrentMembership`, §4.2).

This is why every controller test in the tenancy surface parametrizes over
`access` and `access_v2`: the dual-write guarantee must hold for the entire
request path, not just at token decode.

## §8.5 — Multi-familia switch-account flow

`AccountsMembership.switch_account/2` (`(user, target_membership_id)`) re-issues
the claim set scoped to a *different* membership the same User already holds:

1. Cast `target_membership_id` to UUID (`:membership_not_found` on failure).
2. `load_active_membership/1` — loads the row and requires `status == :active`
   (`:membership_not_active` otherwise; `:membership_not_found` if no row).
3. `assert_owner/2` — the membership's `user_id` must equal the caller's id, else
   `:not_your_membership`.
4. Re-fetch the User from the DB (`Repo.get!`) rather than trusting the
   caller-supplied struct (switch is security-sensitive).
5. Return `{:ok, %{user, account, membership, claims}}` where `claims` comes from
   `build_response_claims/3` (flag-gated, §3.3).

`AccountLifecycleController.switch_account` maps errors to HTTP:
`403 not_your_membership`, `409 membership_not_active`,
`404 membership_not_found`. Because there is no `:account_id` in the URL, the
route uses only `:auth` (not `EnforceAccountScope`).

End-to-end sequence (owner invites, invitee accepts, invitee later switches):

```
Owner(A) ── POST /api/accounts/A/invites {email} ──▶ invite/3
             (owner check, seat cap under FOR UPDATE, mint+hash+store token)
          ◀── 201 {invite: {token, expires_at, membership_id}}

Invitee ── POST /api/invites/:token/accept {} | {name,password} ──▶ accept_invite/2
             (verify token, flip :invited→:active, joined_at, claims_for → access_v2 for A)
          ◀── 200 {access_token, refresh_token, user, account, membership, ...}

Invitee (now holds a 2nd membership, Account B)
        ── POST /api/auth/switch-account {membership_id: <B>} ──▶ switch_account/2
             (ownership check, :active check, claims_for → access_v2 for B)
          ◀── 200 {access_token(account: B), account: B, ...}
```

Roster/remove/leave complete the surface: `list_memberships/1` returns
`:active + :invited` rows owner-first (CASE on role, then `joined_at`, then
`inserted_at`) with `:user` preloaded; `remove_member/3` is an owner-only **hard
delete** (Q3) refusing the owner; `leave/2` is a member-only hard delete refusing
the owner. `leave/2` looks the row up by `user_id + account_id` (not `actor.id`)
and checks `:not_a_member` before `:cannot_leave_owned_account` (§11,
divergence 5).

## §9 — Feature-flag rollout & rollback

### §9.1 — Env-var cutover procedure

The cutover is controlled by the application env key
**`:tenancy_v2_only`** (namespace `:meal_planner_api`), read at call time via
`Application.get_env(:meal_planner_api, :tenancy_v2_only, false)` (fail-closed
default `false`). The intended operator-facing control is the environment
variable **`MEAL_PLANNER_TENANCY_V2`**, documented in `ARCHITECTURE.md` and
`docs/FRONTEND_INTEGRATION.md`.

Rollout sequence (matches the three-PR chain):

1. **PR 1 deploy** — `MEAL_PLANNER_TENANCY_V2=false`. Data model + dual-write
   Guardian land; `access_v1` is still the only minted type.
2. **PR 2 deploy** — unchanged (`false`). Application layer lands; controllers
   still read the legacy path.
3. **PR 3 deploy** — deployed with the flag **still `false`**, then **flipped to
   `true` as a separate operation after the deploy completes**. The flip is a
   pure config/env change: **no DB migration, no app release**. Existing
   `access_v1` tokens keep verifying until their normal TTL expires; there is no
   forced re-login. A follow-up change (`tenancy-v2-hardening`) later removes
   `access_v1` *issuance* once the client fully consumes `current_membership`.

What the flag gates (issuance only — verification always accepts both types):

- `AuthController.password/2` (register + login), and
  `Accounts.authenticate_with_password/1`;
- `AccountsMembership.accept_invite/2` and `AccountsMembership.switch_account/2`
  (via `build_response_claims/3`).

`AuthController.refresh/2` never consults the flag — it preserves the incoming
token's `typ` (§3.3).

The env→config binding lives in `config/runtime.exs` (guarded to non-`:test`
environments, since tests set `:tenancy_v2_only` directly via
`Application.put_env`):

```elixir
if config_env() != :test do
  config :meal_planner_api,
         :tenancy_v2_only,
         System.get_env("MEAL_PLANNER_TENANCY_V2", "")
         |> String.trim()
         |> String.downcase()
         |> Kernel.in(["true", "1", "yes", "on"])
end
```

The parse is case- and whitespace-tolerant and fail-closed: any value outside
the truthy set (unset, `false`, `0`, `no`, `off`, garbage) resolves to `false`,
so a mistyped operator value never silently enables v2. This binding was added
by the `tenancy-v2-flag-wiring` change (see §11, divergence 2) after the original
Phase-A branches landed without it.

### §9.4 — Catastrophic rollback

Per-migration `down/0` functions are intentionally **non-destructive** and do not
attempt to reverse data:

- `alter_accounts_to_plan_enum` down re-adds `account_type` + CHECK but does not
  remap data (a `plan → account_type` remap would corrupt `:family_6` accounts).
- `make_user_account_id_nullable` down re-imposes `NOT NULL` (would block if any
  `account_id` were nulled meanwhile).
- `add_account_memberships_backfill` down drops the invariant function but leaves
  backfilled rows in place.

Therefore a true production rollback is a **coordinated restore from the
pre-Phase-A migration snapshot**, not a chain of `mix ecto.rollback`. Because
the flag defaults to `false` and both token types verify, an emergency
"soft rollback" of behavior is simply flipping `MEAL_PLANNER_TENANCY_V2` back to
`false` — minted tokens revert to `access_v1` immediately without any DB change.

## §10 — ADR-style decisions (Q1–Q10)

### Q1 — Guardian JWT claims: legacy `access` vs membership-scoped `access_v2`

**Decision.** Two claim shapes coexist (§3.1, §3.2). The legacy `access`
(`access_v1`) token is account-scoped via `account_id` (+ compat
`account_type`); the `access_v2` token is membership-scoped and self-describing
(`membership_id`, `account_id`, `role`, `plan`, `status`). Guardian verifies both
at all times; the flag controls which is minted.

**As-built refinement (rationale).** The original design resolved the legacy
token's `current_membership` by *synthesizing* an in-memory
`%AccountMembership{__synthesized__: true, id: nil, status: :active}` from
`user.account_id` + `user.role`, with no DB read. This was later replaced (tenancy
debt cleanup / post-PR-3b BLOCKER fix) with a **mandatory real `:active` row
lookup** for `(user.id, account_id)`; on miss the request is denied with
`401 membership_id_required`. Reason: `remove_member/3` and `leave/2` hard-delete
the row without clearing `user.account_id`, and legacy tokens carry a ~4-week TTL
with no server-side revocation — the synthesized path let a removed member keep
full access until the token expired. There is **no `__synthesized__` marker
anywhere in the shipped code**. (See §4.2, §11 divergence 1.)

### Q2 — Backfill invariants enforced in-transaction

The backfill and `check_account_membership_invariants()` run in the same
migration transaction; any violation rolls the migration back so the DB never
persists a broken invariant (§2.4, §2.5).

### Q3 — Membership removal is a hard delete

`remove_member/3` and `leave/2` `Repo.delete_all` the row (no soft-delete /
tombstone). Seat cap is re-checked on the next re-invitation. Plan caps
themselves resolve through `subscription_plans` by `name` (decision 5.3).

### Q4 — Refresh preserves the original `typ`

`refresh/2` reissues whatever `typ` the incoming refresh token implied
(`membership_id` present → `access_v2`, else `access_v1`), independent of the
flag — no silent re-scoping in either direction (§3.3).

### Q5 — Accept both token types without forced re-login

`VerifyTokenType` accepts `access` and `access_v2`; unknown `typ` →
`401 unsupported_token_type`. VerifyHeader carries no `typ` filter because it can
only match one value (§4.1, decision 5.5).

### Q6 — Multi-familia factory helper

`user_with_memberships/2` builds a User with N accounts + N `:active`
memberships; `issue_access_v2_token/2` mints a v2 JWT via the canonical claim
builder (§8.2).

### Q7 — Invite token model

32 random bytes → URL-safe base64 no-padding plaintext (~43 chars); stored as
SHA-256 lower-hex (64 chars); 7-day TTL; single-use enforced by
`status != :invited` (hash retained on the row, §6.1 / §11 divergence 4).

### Q8 — Socket membership resolution

`LoadCurrentMembershipSocket.membership_from_socket/1` resolves the socket's
membership; the socket *connect* succeeds even for a non-`:active` membership so
each channel `join/3` can return a specific `{:error, %{reason: "forbidden"}}`
after its own `status != :active` check (§7). This is why
`load_membership_by_id/2` does not filter status.

### Q9 — `users.role` kept for the dual-write window

`users.role` remains during Phase A (feeds the backfill role default and legacy
compatibility). Dropping it and making `account_memberships.role` the sole source
of truth is a deferred follow-up (§2.3).

### Q10 — `family_6` / `trial` plans + `subscription_plans` resolution

Two new plans are seeded (`family_6` and `trial`, both `max_users: 6`,
`max_planning_days: 30`), and `policy_for_account/1` resolves caps through
`subscription_plans` by `name` rather than hard-coding (§2.2, §2.6).

## §11 — Divergences from `tasks.md` (as-built)

The following are places where the shipped code intentionally differs from the
`tasks.md` descriptions. The code is authoritative.

1. **No synthesized legacy membership (security fix).** `tasks.md` (tasks 1.10,
   2.2; Q1) describes synthesizing an in-memory
   `%AccountMembership{__synthesized__: true, id: nil, status: :active}` for
   legacy `access` tokens with no DB read. The as-built `LoadCurrentMembership`,
   `LoadCurrentMembershipSocket`, and `AccountsMembership.current_membership/2`
   all **require a real `:active` row** and deny with `401 membership_id_required`
   on miss. No `__synthesized__` marker exists in the code. This closed a stale
   removed-member access hole.

2. **Cutover env-var wiring landed separately.** The original three Phase-A
   branches shipped the flag *reader*
   (`Application.get_env(:meal_planner_api, :tenancy_v2_only, false)`) but never
   bound `MEAL_PLANNER_TENANCY_V2` to `:tenancy_v2_only`, so the flag defaulted to
   `false` in every deployed environment and the documented cutover could not
   actually be performed — it was only flippable in tests via
   `Application.put_env`. The env→config parse (trim/downcase, truthy set
   `true/1/yes/on`, fail-closed) was subsequently added to `config/runtime.exs`
   by the `tenancy-v2-flag-wiring` change (§9.1), closing the gap. The design
   above documents the wired state.

3. **`accept_invite/2` and `switch_account/2` honor the flag.** `tasks.md`
   (tasks 2.4, 2.8) has them calling `claims_for/2` unconditionally. As-built they
   route through `build_response_claims/3`, which mints `access_v2` only when the
   flag is on and legacy `Accounts.claims_for/2` otherwise — making the flag a
   real killswitch for these two flows (post-review fix).

4. **Invite token columns retained on accept.** `tasks.md` (tasks 2.4, 2.7) says
   accept nulls `invite_token_hash` and `invite_expires_at`. As-built the row
   **keeps** both; replay is detected by `status != :invited` returning
   `:invite_token_used`. Only `status`, `joined_at`, and `user_id` change on
   accept.

5. **`leave/2` looks up by `user_id + account_id`.** Not by `actor.id` (which
   could be `nil` for a synthesized struct historically), and it checks
   `:not_a_member` before `:cannot_leave_owned_account` so a User who owns a
   *different* account gets `:not_a_member` rather than a misleading
   `:cannot_leave_owned_account`.

6. **Only four channels exist.** The channel sweep covers `CalendarChannel`,
   `PlanningChannel`, `CookingChannel`, `AiChannel`. The `shopping`/`inventory`
   channels referenced in planning were never created; those domains enforce
   isolation via `current_membership.account_id` in their HTTP controllers.

7. **`VerifyTokenType` is a standalone plug.** Rather than "an additional
   VerifyHeader step" (task 1.11 wording), the `typ` filter was dropped from
   `VerifyHeader` entirely and a dedicated `VerifyTokenType` plug enforces the
   supported set — the only way to accept two `typ` values through Guardian.

8. **Guardian still re-attaches `:subscription_tier` and `:account_id`.** Only
   `:account_type` reattachment was removed. The other two remain because
   controllers still read `user.subscription_tier` and the legacy membership
   lookup needs `user.account_id`; their removal is a later follow-up.

9. **`policy_for_account/1` landed early.** `subscriptions.ex` already read
   `Account.plan` after PR 1 (a forward-compatible preview); the nominal PR 2
   task became test coverage + docstring rather than a production rewrite.
